# Подводные камни и смягчение

## 1. etcd-кворум с домашним сервером

- **Риск:** домашний интернет-канал нестабилен; флапы → переизбрание лидера etcd, кратковременные
  подвисания control-plane.
- **Смягчение:** 2 VPS держат кворум при флапе дома; падение любого одного из 3 узлов переживается.
  При желании настроить так, чтобы лидер предпочтительно жил на VPS.
- **План Б:** если домашний канал совсем плохой — вынести дом из кворума (сделать agent-нодой) и
  взять 3-й VPS в server-роль. Кворум всегда нечётный ≥3.

## 2. k3s поверх tailnet за NAT

- **Риск:** без явной привязки ноды анонсируют приватные LAN-адреса и не сходятся.
- **Смягчение:** при установке задавать `--node-ip=<tailnet-ip>` и `--flannel-iface=tailscale0`
  на каждой ноде; `--tls-san` с tailnet-адресом первой ноды.

## 3. Durable требует дисциплины (идемпотентность)

- **Риск:** при повторном выполнении durable-шага побочные эффекты дублируются (двойной `git push`,
  двойная отправка, двойная запись).
- **Смягчение:** побочные эффекты только через журналируемые шаги Restate; операции идемпотентны
  (upsert, ключи дедупликации). Именно поэтому durable включаем **только для критичных задач**
  (ADR-004), а не для всего.

## 4. MinIO distributed

- **Риск:** distributed-режим хочет ≥4 диска (erasure coding); на 3 узлах раскладка нетривиальна.
- **Смягчение:** на старте допустим single-node MinIO + регулярный бэкап; апгрейд в distributed —
  отдельным шагом, когда конфигурация дисков ясна.

## 5. Латентность VPS ↔ дом

- **Риск:** RTT дом↔VPS (десятки мс) влияет на отзывчивость etcd и репликации Postgres.
- **Смягчение:** брать VPS географически ближе к дому; мониторить задержки кворума; для Postgres —
  синхронная репликация только если латентность позволяет, иначе асинхронная.

## 6. Незрелость дашборда (Mission Control)

- **Риск:** очень свежий проект (взрывной рост в 2026), API/поведение могут меняться.
- **Смягчение:** дашборд = сменный модуль (ADR-007). Не завязывать на него ядро; иметь Restate UI как
  запасной интерфейс.

## 7. Безопасность

- **Риск:** API-ключи (Claude, OpenRouter), пароли Postgres, доступ к кластеру.
- **Смягчение:**
  - Секреты — SOPS/age + k8s Secrets; не plaintext в манифестах и не в git.
  - Доступ к сервисам — только внутри tailnet; tailnet ACL ограничивают, какие устройства куда ходят.
  - kube-apiserver не публиковать в интернет — только через tailnet.
  - Ollama-эндпоинты на Mac/Windows — слушать на tailnet-интерфейсе, не на 0.0.0.0.

## 8. Воркеры на Windows/Mac

- **Риск:** соблазн сделать их нодами кластера.
- **Смягчение:** они **только** agent-воркеры, цепляются к Restate/LiteLLM по сети. В etcd-кворум их
  не включать — выключение устройства не должно влиять на кластер (ADR-003, ADR-005).

## 9. Слабые удалённые VPS

- **Риск:** бюджетные VPS 1 vCPU / 1 ГБ RAM / 15 ГБ SSD могут не выдержать k3s server-node с embedded
  etcd даже без пользовательских workload: OOM, медленный etcd, долгие рестарты, нестабильный quorum.
- **Смягчение:** удалённые VPS использовать как тонкие control-plane/etcd-ноды, не планировать на них
  тяжёлые приложения, отключить лишние packaged components, ограничить workload на домашнем сервере,
  мониторить RAM/disk/etcd latency.
- **План Б:** если Этап 1 не проходит веху, не двигаться дальше. Владелец выбирает: увеличить VPS,
  заменить топологию на 3 более подходящих server-ноды или честно принять режим recovery вместо HA.

## 10. Разрастание third-party controllers и CRD

- **Риск:** каждый новый controller/CRD/Helm chart добавляет RBAC, новые образы, API surface, обновления,
  CVE и нагрузку на слабые VPS. Для ИИ это также больше контекста и больше способов ошибиться.
- **Смягчение:** dependency budget: ставить только то, что нужно для текущей вехи. Перед добавлением
  компонента фиксировать: зачем он нужен, какие ресурсы потребляет, какие права требует, как обновляется,
  как удаляется и как тестируется.
- **План Б:** если компонент не критичен, заменить runbook/script-решением или отложить до отдельного
  этапа. Тяжёлые наборы вроде Longhorn/Rook/kube-prometheus-stack не ставить на старте.

## 11. Hysteria Realms как ложный сетевой фундамент

- **Риск:** Hysteria Realms выглядит привлекательным из-за NAT traversal, но это proxy/hole-punching
  механизм, а не полноценная identity-aware mesh-сеть для k3s/etcd. Некоторые NAT-сценарии не пробиваются.
- **Смягчение:** Tailscale остаётся underlay для всех критичных связей. Hysteria Realms использовать
  только как аварийный/экспериментальный доступ, не включать в путь quorum/API/Restate/LiteLLM.

## 12. Неправильное разделение Go и Rust

- **Риск:** если писать всё на Rust, можно замедлить реализацию и усложнить работу ИИ-агентов; если писать
  всё на Go, можно упустить преимущества Rust для долгоживущих daemon/proxy.
- **Смягчение:** Go-first для Kubernetes/GitOps glue, CLI, health-checks и первых durable-воркеров.
  Rust — для автономных системных компонентов, где важны memory safety, низкий overhead и предсказуемое
  поведение. Python — только для одноразовых скриптов и анализа.

## 13. Local-first bootstrap принимают за HA

- **Риск:** старт на одном домашнем Linux-сервере полезен, но не даёт отказоустойчивости. Если забыть эту
  границу, можно начать ставить критичные сервисы без quorum/failover и получить ложное ощущение готовности.
- **Смягчение:** явно маркировать статус как `local bootstrap`; не закрывать Этап 1 без двух VPS и
  failover-вехи; все runbooks писать так, чтобы они переносились в будущий 3-node профиль.

## 14. Две локальные Proxmox-ноды принимают за защиту от отказа площадки

- **Риск:** `uap-home-1` и `uap-home-2` на разных Proxmox-нодах улучшают локальную живучесть, но общий
  дом/роутер/электричество/интернет остаются shared failure domain.
- **Смягчение:** документировать топологию как `2 local + 1 remote`, а не как защиту от полного отказа
  дома. Для защиты от полного локального отказа нужна целевая топология `1 local + 2 remote` или отдельная
  удалённая площадка с quorum majority.
