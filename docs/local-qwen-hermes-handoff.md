# Локальный Qwen + Hermes — полная документация настроек (handoff)

Всё, что мы настроили, чтобы Hermes-агент работал на ЛОКАЛЬНЫХ моделях (когда платные лимиты
Claude/Codex кончились), со всеми граблями и их фиксами. Собрано из живой сессии 2026-07-06.

> **Про твой сетап:** у тебя Hermes крутится ЛОКАЛЬНО, Qwen — на той же машине. Это ПРОЩЕ нашего
> (у нас Hermes жил в k3s-поде и ходил к моделям на других железках через egress-прокси/tailnet).
> Разделы, помеченные **[наш кластер]**, тебе в основном не нужны — но описаны для полноты и на
> случай, если у тебя тоже под/прокси. То, что помечено **[тебе нужно]** — обязательное.

## 0. Архитектура (что к чему)

```
             ┌─────────── твоя локальная машина ───────────┐
   Hermes ──▶│  llama.cpp :8080  ← Qwen-AgentWorld-35B GGUF │  (мозг)
   (мозг+    │  (localhost, никакого прокси)                 │
   тулзы)    └───────────────────────────────────────────────┘

Наш (более сложный) вариант для справки:
   Hermes-pod (k3s) ──▶ router (ops-1 :8090) ──▶ qwen (desktop :8080, tailnet)
                                              └─▶ ornith (mac :8080, tailnet)
```

Роли по нашим замерам: **Qwen-35B = мозг/reasoning/FC/retrieval**, **Ornith-9B = кодер**.

## 1. Qwen-AgentWorld-35B на llama.cpp **[тебе нужно]**

### 1.1 Точная команда запуска (не импровизируй с флагами)
```
llama-server.exe -m <path>\Qwen-AgentWorld-35B-A3B-UD-IQ4_NL.gguf ^
  --host 0.0.0.0 --port 8080 ^
  --no-mmap -ngl 999 --n-cpu-moe 16 -c 65536 --jinja -fa on
```
- Даёт **~65 tok/s, VRAM ~12.8 GB, RAM ~7 GB** на **RTX 5060 Ti 16GB** (llama.cpp build b9395+ CUDA).
- Температура генерации: **0.6** (рекомендованная для этой модели).
- GGUF ~18 GB (у нас `C:\Users\x3d_mutant\Downloads\Qwen-AgentWorld-35B-A3B-UD-IQ4_NL.gguf`).

### 1.2 Зачем каждый флаг (важно, не убирай)
| Флаг | Зачем |
|---|---|
| `--jinja` | **КРИТИЧНО.** Включает нативный function-calling (`tool_calls`). БЕЗ него тулзы возвращаются как обычный текст → агент «слепнет». Флаг №1 по важности. |
| `--no-mmap` | режет RAM 17→7 GB (веса не дублируются в RAM), скорость та же |
| `-ngl 999` | все слои на GPU |
| `--n-cpu-moe 16` | рычаг VRAM↔RAM (модель — MoE, ~40 экспертных слоёв). Поднимай для длинного контекста. **ОБРЫВ: при ncm≤4** VRAM переполняется → спилл в RAM → скорость падает до ~21 tok/s |
| `-fa on` | flash-attention |
| `-c 65536` | контекст 64k (интерактивный sweet spot) |

### 1.3 Гочи llama.cpp
- **ollama НЕ грузит эту модель** (arch `Qwen3_5` не поддержан) — ТОЛЬКО llama.cpp.
- llama.cpp **ИГНОРИРУЕТ поле `model`** в запросе — отдаёт загруженную gguf. Любое имя модели ок.
- Проверить, что жив: `curl http://127.0.0.1:8080/v1/models`.

### 1.4 ⚠️ ГЛАВНАЯ ЛОВУШКА — thinking-модель
Qwen-AgentWorld МНОГО думает (тяжёлый `<think>`). llama.cpp по умолчанию
(`--reasoning-format deepseek/auto`) кладёт `<think>…</think>` в **отдельное поле `reasoning_content`**,
а `content` остаётся **ПУСТЫМ**, пока модель не закончит думать.

**Следствие:** при маленьком `max_tokens` модель тратит весь бюджет на размышления и НЕ выдаёт
финальный `content` → **пустой ответ**. Выглядит как «модель молчит / thinking сломан».
- Проверено: `max_tokens=800` → content пустой (`finish_reason=length`); `max_tokens=3000` → верный
  ответ + рассуждение в `reasoning_content`.
- **ЛЕЧЕНИЕ:** держи `max_tokens ≥ 3000-4000` в любом клиенте, что бьёт напрямую в модель.
  (Hermes по умолчанию шлёт `max_tokens=65536` — ему норм; ловушка бьёт при **ручных curl-тестах**
  и клиентах с малым лимитом.)
- В коде ответа: бери `message.content`; если пуст — фолбэк на `message.reasoning_content`.
- Хочешь видеть размышления В САМОМ ответе — сервер с `--reasoning-format none` (тогда `<think>`
  остаётся в `content`). Но qwen думает по 1000-2300 символов/ответ — чат станет шумным. Для агента
  скрытый thinking обычно удобнее.

### 1.5 Длинный контекст (128k) — если понадобится
```
-ngl 999 --n-cpu-moe 20 -c 131072 --jinja --no-mmap -fa on ^
  --rope-scaling yarn --yarn-orig-ctx 32768 -ctk q4_0 -ctv q4_0     (temp 0.6)
```
- KV-квант: **q4_0 рабочий; q5_1 НЕ бери** (нет FA-кернела, стопорится >8k = «зависает»).
- Цена длинного контекста — **ВРЕМЯ** (prefill линеен): 128k ≈ 17 мин только на ingest,
  256k ≈ 54 мин. **32-64k — интерактивный sweet spot** (prefill 3-7 мин, 13-17 tok/s).

### 1.6 Управление GPU (вкл/выкл, чтобы освобождать карту)
У нас 3 скрипта в `tools/local-models/`:
- `start-qwen.bat` — запускает сервер (команда из 1.1).
- `stop-qwen.bat` — `taskkill /F /IM llama-server.exe` (освобождает ~13 GB VRAM).
- `qwen-toggle.ps1` — **одна кнопка**: работает→стоп / стоит→старт, с авто-исчезающим popup
  (`WScript.Popup`, ноль зависимостей). Повешен на ярлык рабочего стола (запуск скрыто:
  `powershell -ExecutionPolicy Bypass -WindowStyle Hidden -File qwen-toggle.ps1`).

## 2. Ornith-1.0-9B (mac, mlx_lm.server) — кодер/фолбэк **[опционально]**

_Нужно, только если у тебя есть Mac и ты хочешь второй локальный кодер. Иначе пропусти._

### 2.1 Запуск
```bash
~/mlxenv/bin/mlx_lm.server --model mlx-community/Ornith-1.0-9B-4bit \
  --host 0.0.0.0 --port 8080 \
  --prefill-step-size 512 --prompt-cache-size 1 --prompt-concurrency 1 --prompt-cache-bytes 3000000000
```
- Ornith-1.0-9B-4bit — код-специалист (SWE-bench 69.4), arch Qwen3_5, native FC. ~13-18 tok/s на M4.
- env: `~/mlxenv` (Python 3.12, mlx-lm). Для большого контекста: `sudo sysctl iogpu.wired_limit_mb=14336`.

### 2.2 ⚠️ КРИТИЧНАЯ ГОЧА — OOM без cache-флагов
`mlx_lm.server` **не умеет квантовать KV** и **копит prompt-cache** (видели 10 sequences = 6.34 GB).
На 16GB это даёт `[METAL] Insufficient Memory → libc++abi: terminating` — **C++ abort убивает процесс**
(не ловится из Python).
- **ОБЯЗАТЕЛЬНО** `--prompt-cache-size 1 --prompt-concurrency 1` (убивает накопление кэша — это и было
  разницей между «влезло» и OOM).
- Плюс держи контекст+генерацию под ~48k токенов fp16 (ctx+gen у потолка = OOM).
- Гоняй сервер в **авто-рестарт лупе** (`while true; do mlx_lm.server …; sleep 2; done`), чтобы крэш
  само-лечился.
- mlx_lm.server требует **ТОЧНЫЙ repo-id** (`mlx-community/Ornith-1.0-9B-4bit`), не «local» → иначе 401.
- mlx_lm.stream_generate НЕ применяет chat-шаблон (качество кода падает) — но у СЕРВЕРА шаблон есть, ок.

## 3. Единый роутер (route.py) **[наш кластер — тебе НЕ нужен]**

_Нужен, только если у тебя НЕСКОЛЬКО моделей на РАЗНЫХ железках за одним эндпоинтом. У тебя всё
локально → Hermes бьёт напрямую в `localhost:8080`, роутер не нужен. Описано для полноты._

- `tools/local-models/route.py` — ~60-строчный **stdlib** реверс-прокси (ноль зависимостей),
  systemd-сервис `local-model-router` на ops-1, порт 8090.
- Маршрут по имени: `"ornith"` в имени → мак; иначе → qwen (дефолт). **Перезаписывает `model` под
  бэкенд** (mlx нужен точный id, llama.cpp игнорит). Стримит ответ → нативный `tool_calls` проходит.
- **Fast-fail:** connect-timeout **10с** (было 1200) — иначе на выключенном бэкенде роутер ВИСНЕТ, и
  Hermes-fallback не срабатывает. Read-timeout 1200с (медленная генерация thinking-модели).
- `ProxyHandler({})` — форсит no-proxy (роутер локальный).
- Эндпоинт: `http://100.82.241.121:8090/v1` (tailnet) / `http://192.168.0.203:8090/v1` (LAN).
- LiteLLM в кластере НЕ подошёл гейтвеем: **поды k3s не маршрутизируются в tailnet 100.x** (pod-network
  разрыв). ops-1 (always-on, Linux, tailnet) достаёт оба бокса — поэтому роутер там.

## 4. Мозг Hermes → локальная модель **[тебе нужно, но проще]**

### 4.1 Конфиг модели (hermes-agent 0.18.x)
```yaml
model:
  provider: custom
  base_url: http://127.0.0.1:8080/v1      # локально = localhost; у нас был роутер (100.82.241.121:8090/v1)
  default: qwen-35b                        # поле ИМЕНИ = `default` (НЕ `model`!); продублируй и в `model`
  model: qwen-35b
  api_key: local                           # любая непустая строка; llama.cpp ключ игнорит
```
Опционально фолбэк (если модель отвалится):
```yaml
fallback_providers:
  - provider: custom
    base_url: http://127.0.0.1:8080/v1
    default: <вторая-модель>
    model: <вторая-модель>
    api_key: local
```

### 4.2 Схема провайдера — точные ключи (мы это раскопали)
- `provider: custom` = generic OpenAI-совместимый эндпоинт.
- Ключ URL — **`base_url`** (НЕ `api_base`, НЕ `openai_base_url`).
- Поле имени модели — **`default`** (в отличие от codex, где было `model`). Ставь ОБА для надёжности.
- `openai_runtime: codex_app_server` — это codex-специфика, для `custom` **убери**.
- Гард hermes при старте проверяет наличие `model.provider` (иначе pod в CrashLoop — если у тебя pod).
- **FC на клиенте включается автоматически** (Hermes шлёт `tools[]`), но РЕАЛЬНО `tool_calls` вернутся
  ТОЛЬКО если сервер запущен с `--jinja` (llama.cpp). Это главное условие FC.

### 4.3 «Reasoning: off» — это НЕ баг
`hermes config` для custom-провайдера показывает `Reasoning: off` — норма. Hermes не распознаёт
кастомную модель как reasoning-типа и не рендерит трейс. Модель при этом ДУМАЕТ (reasoning_content
полон). Мы пытались включить перебором ключей — не помогает; поле дисплейное. Хочешь видеть
размышления — `--reasoning-format none` на сервере (см. 1.4), а не в конфиге Hermes.

### 4.4 ⚠️ /compress (сжатие контекста) на медленной локальной модели — ГОЧА
`/compress` (и авто-сжатие при заполнении контекста) просит **сам мозг** просуммировать большой кусок
диалога. На локальном Qwen это медленно: prefill 62K токенов ≈ 8 мин + thinking + генерация саммари →
**15+ минут**. А дефолтный таймаут `auxiliary.compression.timeout = 120с` → **`Compression aborted:
Request timed out`** (диалог не теряется, но не сжимается).
- Симптом: `/compress` висит и падает по таймауту; статус-строка «Approx request size ~62,402 tokens».
- **ФИКС** (soft-config, `hermes config set …`, переживает рестарт; после — рестарт gateway):
  ```
  hermes config set auxiliary.compression.timeout 900   # было 120с; дай медленной модели время
  hermes config set compression.threshold 0.2           # было 0.5→0.3→0.2; сжимай РАНЬШЕ (на ~20% контекста)
  hermes config set compression.target_ratio 0.15       # было 0.2; саммари меньше -> генерится быстрее
  ```
  Логика: для медленной локальной модели **частые маленькие сжатия лучше редких огромных**. Ниже порог
  = живой контекст держится ЛЕГЧЕ → и ответы мозга быстрее, и каждая компакция мельче/быстрее. Таймаут
  держи < read-timeout роутера (у нас 1200с), если ходишь через роутер. **(2026-07-08 снизили 0.3→0.2.)**
- **НЕ уводи компакцию на вторую (меньшую) модель — замерено, не помогает.** Соблазн: `auxiliary.compression.model`
  = дешёвая модель (в Hermes есть `hermes models` ровно для этого, «−85% фоновых токенов»). Но у нас замер
  (27K токенов на ornith-9b/mac через роутер) = **179с ≈ как qwen (не быстрее)** + вернулось **ПУСТОЕ содержимое**
  (thinking-модель спалила лимит на рассуждения → саммари было бы пустым = ПОТЕРЯ контекста). Компакция всё равно
  БЛОКИРУЮЩАЯ (ждёшь в любом случае), обе модели локальные/бесплатные (экономии токенов, ради чего это в Hermes,
  нет). Итог: оставили компакцию на основном мозге (`Model: (auto)`). Реальный рычаг — ЛЁГКИЙ контекст (порог), а
  не смена суммаризатора. Ключи: `auxiliary.compression.model`, `compression.protect_last_n` (20); видно в
  `hermes config show` → «◆ Context Compression».
- У тебя локально (без роутера) — то же самое, только `base_url` = localhost. Или, если контекст пухнет
  быстрее, чем сжимается — начинай свежую сессию (`/new`).

### 4.5 [наш кластер] Развёртывание, если Hermes в k3s
- Мозг GitOps-owned в `managed-config` (ConfigMap → `/etc/hermes/config.yaml`, RO). Меняешь → бампаешь
  аннотацию `hermes-agent/config-rev` в Deployment → Flux reconcile → pod роллится.
- У тебя локальный Hermes → просто правишь `~/.hermes/config.yaml` (`hermes config set …`) и рестарт.

## 5. ⚠️ NO_PROXY / прокси — гочи **[наш кластер; у тебя localhost → НЕ касается]**

_Только если Hermes ходит к модели ЧЕРЕЗ egress-прокси (у нас — да, VLESS из РФ). У тебя всё
локально, прокси нет → пропусти. Но это была наша БОЛЬШАЯ боль, документирую._

- **httpx / urllib ИГНОРИРУЮТ CIDR в `NO_PROXY`.** Мозг-вызовы к `100.82.241.121` через CIDR
  `100.64.0.0/10` уходили в прокси → **502**. Фикс: **ТОЧНЫЙ IP** в NO_PROXY (не CIDR).
  Проверено в поде: CIDR→502, exactIP→200. (Тот же корень, что в offload.py: «Python ignores NO_PROXY CIDR».)
  - Симптом до фикса: **tool-call без вывода** («не вижу ответа») — qwen эмитит tool_call, а СЛЕДУЮЩИЙ
    brain-вызов (обработать результат) падает 502 → ответа нет. Это НЕ сломанный тул-цикл (он ок после фикса).
- **curl ЧЕСТНО матчит CIDR** (в отличие от httpx). Поэтому для доступа терминал-тула к LAN (Proxmox
  192.168.0.169 и т.п.) хватило `192.168.0.0/16` в NO_PROXY.
- **Гоча теста:** `kubectl exec … env` показывает env КОНТЕЙНЕРА (старый NO_PROXY), а терминал-тул
  наследует `os.environ` процесса hermes (managed `/etc/hermes/.env`). Проверяй доступность через сам
  тул или `set -a; . /etc/hermes/.env; curl …`, иначе ложный 000.

## 6. `ornith` — кодер-тул для мозга **[опционально; см. вывод]**

_Идея: qwen-мозг делегирует НАПИСАНИЕ кода более сильному кодеру (ornith)._

- `tools/local-models/…` → ключ ConfigMap `ornith-wrapper` → сеется в `/opt/data/.local/bin/ornith`.
  Скрипт: `ornith "<задача>"` → curl в роутер (`ornith-9b`), форсит no-proxy, снимает markdown-фенсы →
  чистый код. Правило в USER.md велит мозгу делегировать кодинг ему.
- **ЧЕСТНЫЙ ВЫВОД:** тул РАБОТАЕТ (qwen его вызвал → ornith вернул bubble_sort+тесты, счётчик вызовов
  подтвердил), **НО авто-делегирование НЕНАДЁЖНО** — qwen пишет простой код САМ и **игнорирует даже
  жёсткое правило** «ты код не пишешь», потому что код — его обычный текстовый вывод, а не тул-вызов,
  который можно принудить. **Делегирует, когда просишь ЯВНО** («используй ornith»). Это ограничение
  hermes (нельзя декларативно форсить роутинг кодинга). Вывод: ornith = сильный кодер **по требованию**,
  не форсированный дефолт.
- Для тебя: если Qwen-мозг тебя как кодер устраивает (наш замер 75-96%) — отдельный кодер не нужен.

## 7. Клиент Hermes-chat (как заходить в агент) **[тебе нужно]**

Наш запуск = **Git Bash → `ssh hermes`** в reconnect-цикле; `ssh hermes` идёт на ops-1 и запускает
`~/bin/hermes-chat`. Для локального Hermes тебе проще — запускай `hermes chat` прямо на машине.
Если всё же по ssh на отдельный хост:
```
# ~/.ssh/config
Host hermes
    HostName <IP хоста Hermes>
    User <user>
    RequestTTY yes
    RemoteCommand ~/bin/hermes-chat        # или: hermes chat
    ControlMaster auto
    ControlPath ~/.ssh/cm-%r@%h:%p
    ControlPersist yes
    ServerAliveInterval 20
    ServerAliveCountMax 3
    Ciphers aes128-gcm@openssh.com
```
```bash
# ~/bin/hermes.sh — держать вечно, переподключаться при обрыве
while true; do ssh hermes && break; echo "[reconnect 2s]"; sleep 2; done
```
Нужно: Git for Windows (Git Bash+ssh) + авторизованный ssh-ключ + этот config + hermes.sh.
Красивый ярлык: target `"C:\Program Files\Git\bin\bash.exe" -l -c "~/bin/hermes.sh"` (+ иконка).

## 8. Что модели умеют / где проседают (наш 4-часовой soak, детерминированные грейдеры)

### Qwen-AgentWorld-35B
| Способность | Результат |
|---|---|
| Function-calling | **100%** (single/parallel/distractor/no-call) — надёжный агентский мозг |
| Кодинг | сильный, **~75-96%** (склонен к багам на edge-cases, напр. `gcd(0,7)` может валить каждый раз при temp0) → для боевого кодинга нужен самотест-цикл, не one-shot |
| Retrieval | до ~64k отлично; на большом контексте **через СЕРВЕР теряет «середину»** (у сервера нет kv-quant); дроби длинный контекст |
| Prompt-injection | устойчивость **~80-87%** — НЕ 100%; на недоверенный вход ставь внешний guard |
| Самопроверка кода | ловит **100%** тонких багов, но **~45% ложных** на корректном коде → пары с ИСПОЛНЯЕМЫМИ тестами, не с одиноким суждением |

### Ornith-1.0-9B (если используешь)
кодинг **20/20=100%** (включая gcd — БЬЁТ Qwen-35B), injection 5/5, FC 16/16, retrieval 67%
(теряет середину), offload-заземление 79% (все галлюцинации ловятся грепом). Медленный (~13-18 tok/s).

1. Qwen запускай **СТРОГО с `--jinja`** (иначе тулзы мертвы) + командой из §1.1.
2. **`max_tokens ≥ 3000`** в прямых запросах, иначе пустой `content` (модель ещё думает).
3. Мозг Hermes: `provider: custom`, `base_url: http://127.0.0.1:8080/v1`, `default`+`model`=имя,
   `api_key: local`.
4. Всё локально → **прокси-боли (NO_PROXY/502) у тебя НЕ будет** (это была наша кластер-специфика).
5. Qwen — thinking-модель: `<think>` уходит в `reasoning_content`, `content` = финальный ответ.
6. `Reasoning: off` в hermes config для custom — норма, не баг.
