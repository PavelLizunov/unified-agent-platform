# Ornith-1.0-9B-4bit (Mac M4) — soak-eval (результаты)

_Прогон 2026-07-05/06. Модель = `mlx-community/Ornith-1.0-9B-4bit` (код-специалист, arch Qwen3_5,
SWE-bench Verified 69.4) на **pavels-mac-mini M4 / 16GB** через `mlx_lm.server` (MLX). Грейдер везде
**детерминированный** (греп цитат / скрытые cargo-ассерты на build-1 / exact-match), НЕ суждение модели.
Тот же харнесс `tools/offload/soak/`, что гонял Qwen-AgentWorld-35B — числа **напрямую сравнимы** с
[Qwen V3](../qwen-agentworld/V3_REALDATA_REPORT.md)._

> **STATUS: DONE** — 5 раундов, **4.17 ч модельного времени, 0 крашей на финальной конфигурации**,
> ~45 мин/раунд. Мак **always-on** (подтверждено владельцем) → в отличие от desktop-GPU можно гонять долго.

## Как гоняли (и почему не с воркстейшена)

- **Хост харнесса = `uap-ops-1` (Linux), НЕ Windows-воркстейшен.** Windows↔мак по tailnet **рвёт долгие
  HTTP/SSH-соединения** (`WinError 10054` на ~34с; и туннель тоже). Короткие команды проходят, генерация
  thinking-модели — нет. Linux↔мак держит (72с+ без обрыва). Модельный трафик идёт `ops-1 → мак:8080`,
  грейдер кодинга — `ops-1 → build-1` напрямую.
- **Управление сервером** — durable `ops-1 → мак` SSH (поставили ключ ops-1), сервер в **авто-рестарт лупе**.
- Харнесс сделан **model-agnostic**: `OFFLOAD_MODEL` (mlx_lm.server требует точный repo-id; llama.cpp
  игнорит), `OFFLOAD_MAX_BYTES` / `OFFLOAD_MAX_TOKENS` — лимиты контекста/генерации per-backend.

## Находка №1 (инфра): 16GB + mlx_lm.server OOM-ит на больших контекстах

`mlx_lm.server` **не умеет квантовать KV-кэш**. Первые прогоны падали:
`[METAL] Command buffer execution failed: Insufficient Memory → libc++abi: terminating` — **C++ abort
убивает процесс** (не ловится из Python). Два усугубляющих фактора:
1. **Накопление кэша** — сервер держал **10 sequences = 6.34 GB** KV; именно это было разницей между
   «влезло» и OOM. Фикс: `--prompt-cache-size 1 --prompt-concurrency 1`.
2. **KV дорастает до потолка** — контекст + генерация thinking-модели у ~48k токенов fp16 = OOM.
   Фикс: контекст `OFFLOAD_MAX_BYTES=80000` (~33k-токенный промпт) + генерация `OFFLOAD_MAX_TOKENS=6000`
   → макс KV ~35k, с запасом.

**Итог:** с лимитами — **5 раундов / 4.17ч / 0 крашей**. Но вывод стоит: **offload/retrieval больших
контекстов на 16GB-маке через mlx_lm.server ненадёжен** (в отличие от desktop, где llama.cpp квантует KV
и держит 128k). Плюс медленно: prefill 33k плотного текста ~4 мин, thinking-ген сверху → ~45 мин/раунд.

## Находка №2 (методология): «галлюцинации» offload были артефактом грейдера

Первый прогон дал UNVERIFIED=7/раунд — будто Ornith массово выдумывает цитаты. **Проверка источников:
все 7 найдены дословно.** Ornith (chat/код-модель) «причёсывает» вывод — снимает markdown (`**bold**`,
`` `code` ``) и добавляет `...`-эллипсис, а `verify_quote` сравнивал байт-в-байт. Починил `norm()`
(снимать `*`/`` ` `` с обеих сторон + ведущий/замыкающий эллипсис). Проверено: реальные цитаты проходят,
**фабрикованный ключ по-прежнему ловится** (грейдер стал честным, не слепым). UNVERIFIED упал 7→2/раунд.
_(Принцип: подозревать свой харнесс раньше вывода о модели.)_

## Результаты (5 раундов, Wilson 95% CI)

| Трек | Ornith-9B (мак) | Qwen-35B (V3) |
|---|---|---|
| **Кодинг** (скрытые cargo-ассерты, mutation-proof) | **20/20 = 100%** [84%, 100%] | 75% (`gcd` 0/8) |
| **Injection RESISTED** | **5/5 = 100%** [57%, 100%] | 87.5% |
| **FC — реальные tool-call'ы** | **16/16 = 100%** [81%, 100%] | 100% |
| **FC — строго, вкл. no-call** (4 раунда) | 16/20 = 80% [58%, 92%] | 100% |
| **Retrieval** needle 83KB (3 глубины) | 10/15 = 67% [42%, 85%] | 100% |
| **Offload** grounding (verified/цитировано) | 31/39 = 79% [65%, 89%] | 99.3% |
| **Offload** — галлюцинации пойманы грепом | **8/8 = 100%** | 100% |

### Track B — кодинг (специализация, подтверждена)
**20/20, включая `gcd(0,7)` КАЖДЫЙ раунд** — тот самый edge-case, что Qwen-35B проваливал 0/8.
Ornith — код-специалист (SWE-bench 69.4), и это видно на моих скрытых ассертах: `add`/`is_prime`/`reverse`/`gcd`
все зелёные, стабильно. **На кодинге 9B-мак обходит 35B-desktop.**

### Track C — retrieval + injection
- **Retrieval 67%**: игла на `start`/`end` находится всегда, в `middle` — **промахивается каждый раунд**
  («lost in the middle» на 83KB fp16). _(Замечание: [память](../../..) фиксировала retrieval 100% до 256k
  через `mlx_lm generate` с kv-quant; `mlx_lm.server` без kv-quant на fp16 середину теряет.)_
- **Injection 5/5 RESISTED**: на отравленный контекст модель выдаёт легитимное резюме, не маркер. Надёжно.

### Track D — function-calling
**Реальные вызовы 16/16** (single + parallel, имя+аргументы точно). Единственный промах — **no-call**
(«Thanks, that's all!»): Ornith **правильно НЕ вызывает тул** (верное воздержание), но отвечает вежливой
прозой вместо литерального `[]`, которого ждёт промпт-грейдер → UNPARSEABLE. Нативный FC (через `tools`
mlx_lm.server) эмитит корректные `tool_calls`. Промах форматный, не по способности.

### Track A — offload
Grounding 79% (31 verified / 8 UNVERIFIED / 24 честных not_found за 5 раундов). **Все 8 UNVERIFIED пойманы
грепом** (safety-свойство держится 100%), и это в основном формат-остаток. Но offload — **не роль Ornith**:
медленно (~45 мин/раунд) и OOM-склонно на 16GB. not_found всегда ≥4 (честен на негативных контролях, AWS-ключ
не выдумал).

## Связный портрет Ornith-9B

**Сильная суть, вольная дисциплина формата.** Это chat/код-модель, не жёсткий экстрактор: «причёсывает»
markdown, отвечает прозой на abstain, добавляет эллипсис. Отсюда практический вывод — **харнесс под неё
должен быть формат-толерантным или с constrained-выводом**, иначе строгий грейдер занижает реальную способность.

## Вердикт

- ✅ **Лучший локальный КОДЕР из протестированных** — 100% на скрытых ассертах, берёт edge-case, где Qwen-35B
  падал. Для оппортунистического кодинга (когда claude/codex-egress лежит, или сэкономить) — годен, **в
  verify-loop** (память: one-shot ненадёжен из-за 4-bit квант-шума, в цикле сходится за 1-3 итерации).
- ✅ **Injection и реальные FC-вызовы — надёжны** (100%).
- ⚠️ **Retrieval теряет середину**, **offload медленный + OOM-склонный** на 16GB — не роль этой модели.
- ✅ **Стабильна** 4.17ч/5 раундов под нагрузкой (с лимитами контекста).

**Durable-brain остаётся Codex.** Мак теперь always-on (снимает прежнее «not-always-on» ограничение), но
16GB тесноваты для больших контекстов, и Ornith — узкий кодер, не универсал. Роль = **оппортунистический
локальный код-воркер (в verify-loop) + запасной FC**, с известными границами (retrieval-середина, формат-вывод,
OOM на больших входах). Данные подтверждают на реальном 4-часовом прогоне, что подходит именно для этого.
