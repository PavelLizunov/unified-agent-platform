## GOAL — что меряем

Оценить модель как кандидата в brain по трём use-case сразу: **A** кодинг, **B** агент/tool-calling, **C** длинный контекст. Метрики **абсолютные** (через якорь `claude -p` на тех же задачах) и **относительные** (какой конфиг лучший).

**Матрица:**
- **Поведение (A/B/D)** на ref-сервере 32k/f16: sampling `{think-ON: temp 0.6/0.9/0.0, think-OFF: temp 0.7}` × (8 кодинг с `python`-exec грейдером + 5 FC + 4 формат), N=3–5.
- **Якорь**: те же кодинг-задачи через `claude -p` → «планка хорошо».
- **Контекст-лестница (C)**: ctx `{8/16/32/64/96/128k}` × KV `{f16, q8_0, q5_1, q4_0}` × think `{off, on}` — на каждой ячейке один haystack (85% окна) с 6 вопросами: NIAH×4 глубины + 2-hop + агрегация. >32k всегда с YaRN. `ncm` авто-подбор под VRAM.

Детерминированные грейдеры считаются на лету; сырые выводы сохраняются → я оцениваю качество **вслепую** (конфиги перемешаны) пост-фактум.

## LOOP — порядок выполнения

1. Поведение: 4 sampling-конфига (A/B/D) на 32k-сервере.
2. Якорь `claude -p`.
3. Контекст-лестница **think-OFF** (быстрее, ядро C) — все ctx × все KV.
4. Контекст-лестница **think-ON**.

Очередь = приоритет: если прервётся, высокоинформативное уже готово. Ожидаемое время ~5–7 ч (лимита нет). После каждого вызова строка пишется в `results.jsonl` и синкается на диск.

**Resume:** при перезапуске читает готовые `(ячейка, задача, сэмпл)` и пропускает их. Обёртка `run-overnight.ps1` перезапускает харнесс, если процесс умрёт (стоп по маркеру `DONE`, макс 40 попыток).

## HOOK — предохранители (не крашить машину)

Свежий сервер на каждую ячейку, между ячейками — `taskkill /F /T` (дерево).

| Страж | Порог | Действие |
|---|---|---|
| **Pre-flight VRAM** | прогноз > 15100 MiB | ячейка **пропускается** (не запускается) |
| **Runtime VRAM** | `nvidia-smi` > 15600 MiB, 2 замера подряд | убить сервер, `trip`, дальше |
| **RAM** | доступно < 2 ГБ | убить сервер |
| **Зависший prefill** | `/slots` не растёт 150 c | убить (для 128k) |
| **Спилл** | (косвенно — низкий прогресс/таймаут) | total-timeout `300 + ctx/100` c |
| **ncm-авто** | держит прогноз VRAM ≤ 14800 | больше эксперта в RAM → VRAM под KV |

Все срабатывания логируются со **статусом** (`skip`/`trip`/`server_failed`) → в утреннем отчёте видно «пропущено по VRAM» vs «убито на замере».

## Крэш-стойкость результатов

- `results.jsonl` — **append + fsync** после каждой строки. Хард-краш теряет максимум один текущий вызов.
- `raw/` — полный вывод каждого вызова отдельным файлом (переживает краш, нужен для слепой оценки).
- `progress.log` — человекочитаемый ход.
- После полного OS-краша: перезапусти ту же команду — продолжит.
