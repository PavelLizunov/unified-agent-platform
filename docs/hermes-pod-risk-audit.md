# Аудит рисков: агент (qwen-мозг) в крошечном поде Hermes

Систематический разбор ВСЕХ классов проблем вида «браузер в поде -> OOM» — не точечно, а по всем
инструментам/возможностям. Триггер: 2026-07-06 qwen-мозг попытался `apt-get install` + запустить chromium
В ПОДЕ (лимит 1.5 GB) -> OOM (exit 133), что рискует уронить сам gateway.

## Среда (почему риск широкий)
- **Под gateway: лимит 1.5 GB RAM / 2 ядра, ЭФЕМЕРНЫЙ** (ФС контейнера сбрасывается при перекате).
- **В поде УЖЕ стоят тяжёлые бинари:** `chromium`, `ffmpeg`, `gcc`, `make`, `node`, `npm`, `xvfb`/`Xvfb`.
  Их наличие обманчиво — любой запустить в 1.5 GB = OOM. (`cargo` НЕ стоит — Rust-сборки и так на build-1.)
- **Конфиг Hermes: все секции защит `null` = ДЕФОЛТЫ** (нет command_allowlist, approvals, security,
  отключённых toolset'ов). Все тулзы/скиллы включены by-default, явных ограничений нет.
- Есть **build-1** (8 ядер / 16 GB, реальная VM) — туда и должен идти весь тяжёлый/персистентный compute.
- Креды в поде: `CLAUDE_CODE_OAUTH_TOKEN` (лимиты), dashboard-пароль, Telegram (managed .env). Почта/соцсети/
  github-токен — НЕ в поде -> эти скиллы упадут без кредов (как image_gen без FAL_KEY). GitHub — только на build-1.

## Матрица рисков (весь класс)

| # | Класс | Триггеры (тулзы/скиллы) | Чем грозит |
|---|---|---|---|
| 1 | **OOM в поде -> падение gateway** | browser_*/chromium, ffmpeg/media (youtube, songsee, heartmula), сборки (gcc/make/npm), тяжёлый execute_code, mlops (audiocraft, evaluating-llms), data-science (jupyter), клон/греп больших репо (github-скиллы), computer-use (xvfb-десктоп), долгие `process` | **Hermes ляжет ДЛЯ ВСЕХ** (OOM убивает процесс агента) |
| 2 | **Побочки наружу (blast radius)** | email (himalaya), social (xurl/X), smart-home (openhue), productivity (google-workspace: календарь/почта/доки, airtable), github push/PR, Telegram-рассылки, cron | реальные необратимые действия во внешнем мире от имени владельца |
| 3 | **Взрыв контекста** | file/search_files/grep по /opt/hermes, research/media/web с большим выводом, чтение больших файлов | переполнение сессии (был инцидент 564K токенов), компакт-шторм |
| 4 | **Эфемерность / дрейф** | apt/pip install в поде, write_file/состояние в под | работа теряется при перекате; дрейф «неизменяемого» GitOps-пода; раздувание |
| 5 | **Тихий провал / over-claim** | image_generate (нет FAL_KEY), TTS, любой скилл без ключа | врёт что умеет -> лихорадочно доустанавливает -> см. #1/#4 |
| 6 | **Cron/процессы на автомате** | cronjob, process | тяжёлое/побочное повторяется автономно; орфаны копят память |
| 7 | **Разрушительное** | terminal (`rm -rf`, git reset), правка *.sops.yaml/доступов | потеря данных, утечка секретов |

## Меры — СДЕЛАНО (soft, USER.md; live + ConfigMap)
Добавлен раздел «ГАРДРЕЙЛЫ» в профиль мозга (`/opt/data/memories/USER.md` живьём + ConfigMap `user-profile`),
покрывающий все 7 классов: тяжёлый compute -> build-1; побочки -> подтверждение владельца; контекст-гигиена;
эфемерность; честность о возможностях; аккуратность с cron/процессами; ничего разрушительного без согласования.
Плюс: pod->build-1 переведён на tailnet; browser/apt -> build-1 явно.

> ⚠️ Это SOFT-меры (правила в промпте). Мозг может их игнорировать (как было с ornith-делегированием).
> Для НАДЁЖНОСТИ нужны hard-меры ниже.

## Меры — РЕКОМЕНДАЦИИ (hard; на решение владельца)
1. **Структурно лучшее: перенести gateway Hermes на build-1** (8 ядер/16 GB). Тогда «тяжёлый compute»
   идёт локально без OOM-риска, отпадает класс #1 целиком. (Раньше решили «оставить под как есть» — но это
   было до вскрытия риска.)
2. **Поднять лимит памяти пода** (если на узле uap-home-2 есть RAM) — смягчает #1, но браузер всё равно тяжёл.
3. **Config-ограничения Hermes** (сейчас все `null`): `command_allowlist` (белый список shell), `approvals`
   (гейт-подтверждение для побочных тулзов), отключить неиспользуемые тяжёлые toolset'ы/скиллы (mlops,
   computer-use, media) — убирает #1/#2 жёстко. Схему надо уточнить у hermes-agent.
4. **Убрать тяжёлые бинари из образа** (chromium/ffmpeg/xvfb), если браузер/медиа делаются на build-1 —
   тогда мозг физически не сможет их запустить в поде.
5. **Резать память/CPU для тяжёлых subprocess** (ulimit/cgroup) — падает команда, а не весь под.
